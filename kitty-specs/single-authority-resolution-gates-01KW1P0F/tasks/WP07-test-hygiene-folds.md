## Subtask Guidance

### T033 — Frozen-baseline `/tmp` ratchet

Create two new files:

**`tests/architectural/tmp_ratchet_baseline.txt`** — a sorted, newline-separated list of
every test-file path (relative to the repo root) that currently contains the literal string
`/tmp/`. Generate it by running:

```bash
git grep -rl "/tmp/" tests/ | sort
```

on the planning base branch (`design/infra-logic-separation-2173`) and pasting the output
verbatim into the file. This is the **frozen baseline**: the set of already-offending files
that are grandfathered in. Do NOT add the current count as a number — the file IS the
baseline (a sorted file list is more reviewable and drift-proof than a bare integer).

**`tests/architectural/test_no_tmp_paths_in_tests.py`** — the ratchet gate. It must:

1. Read `tmp_ratchet_baseline.txt` to obtain the allowlisted set of grandfathered files.
2. Walk `tests/` recursively for every `.py` file.
3. For each file NOT in the baseline, grep for the literal `/tmp/` (string search, not regex).
4. If any non-baseline file contains `/tmp/`, the gate FAILS with a message naming the
   offending file and the line(s) containing `/tmp/`.
5. Assert a concrete floor: the discovered set of baseline files is non-empty (anti-vacuous;
   the walker must actually find the grandfathered files). The floor value is `> 50`
   (conservative; the live census is ~98).

The gate does NOT remediate existing offenders — it only fails on NEW occurrences in files
not in the baseline. The full `#1842` litter sweep is explicitly out of scope (spec FR-007).

Mark the gate with `@pytest.mark.architectural`.

### T034 — `/tmp` ratchet self-mutation test

Add a `test_ratchet_blocks_new_tmp_literal` function (within
`test_no_tmp_paths_in_tests.py`) that:

1. Creates a **temporary test file** (using `tmp_path` pytest fixture) in a location
   OUTSIDE the `tests/` tree or uses `monkeypatch` to override the file-walk so the gate
   sees a synthetic file — the key point is that the synthetic file is NOT in
   `tmp_ratchet_baseline.txt`.
2. Writes `/tmp/something` into it.
3. Runs the ratchet logic (extracted as a callable helper, not via subprocess) against the
   synthetic file.
4. Asserts that the ratchet **raises** or **returns a failure indicator** — do NOT use
   `subprocess.run(pytest)` for this; extract the scan logic into a testable helper.
5. Reverts (the `tmp_path` fixture handles cleanup automatically).
6. After revert: run the ratchet against the same location and assert it PASSES.

This self-mutation proof satisfies NFR-002 for WP07 and SC-006 in the quickstart. The
injection site must be **structurally distinct from any existing baseline file** — do not
inject into a file already in `tmp_ratchet_baseline.txt`, because the gate would ignore it.

Implementation note: extract the per-file scan logic into a module-level function
`scan_file_for_tmp_literal(path: Path) -> list[int]` that returns the line numbers with
`/tmp/`. The self-mutation test calls this helper directly. This also satisfies the Sonar
complexity and testability constraints (CLAUDE.md).

## Risks

- **Live census drift**: the `/tmp/` baseline count on the branch at implementation time may differ
  from the spec's ~82 or the pre-task census of ~98 — always regenerate the baseline from the live
  branch, never hard-code a number. The baseline IS the file list, not a count.
- **Scope creep (FR-007)**: do NOT remediate any existing `/tmp/` occurrence in `tests/`. The sweep
  belongs to `#1842`; touching it here violates the frozen-baseline contract.
- **Redundant markers (FR-008)**: adding `@pytest.mark.fast` to a file already collected by CI adds
  noise and will mislead the next agent. Only mark files proven excluded by `--collect-only`.
- **Subprocess-based self-mutation**: using `subprocess.run(["pytest", ...])` for the self-mutation
  test creates a fragile and slow test. Extract the scan logic into a callable helper (T033) and test
  it directly (T034).
- **Wrong file location for T036 record**: the FR-008 verification record must be findable by a
  reviewer; a comment buried in a test function body is insufficient. Use a module docstring or a
  dedicated `.txt` file.
