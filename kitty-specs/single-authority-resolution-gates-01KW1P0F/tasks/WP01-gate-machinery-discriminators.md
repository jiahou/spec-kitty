## Objective

Build the shared AST-gate module and its two discriminators — canonicalizer (def-use provenance) and coord-authority (write-vs-read) — seeded with the complete pre-sweep baseline allowlist so the gate lands GREEN. This is the foundation both lane A and lane B consume; no other WP can start until this one is done.

## Context

The spec (`kitty-specs/single-authority-resolution-gates-01KW1P0F/spec.md`) identifies two architectural violations that currently have no CI enforcement: (1) un-canonicalized handles reaching `primary_feature_dir_for_mission` and (2) mission-artifact writes resolved via the kind-blind `resolve_feature_dir_for_mission` where the kind-aware authority is mandated. The plan (`plan.md`) and contracts (`contracts/resolution-gates.md`) specify two AST discriminators sharing one gate module, with composite-keyed allowlists and concrete integer floors. The data model (`data-model.md`) defines the boundary objects and their invariants.

The existing Idiom-B machinery in `tests/architectural/surface_resolution_audit/audit.py` and `test_single_mission_surface_resolver.py` provides the scan-shape to reuse (C-005). Both precedents use simpler key schemes — `audit.py` keys on raw `rel_path:line`, `test_protection_resolver_call_sites.py` uses a bare module frozenset — so the composite-key `(enclosing_qualname, token_line)` required by NFR-001 is net-new code. Do not copy these verbatim; study them as structural reference only.

**Merge-blocker constraints active in this WP:**
- C-001: `_read_path_resolver.py:454` is the sanctioned bare probe inside the canonical fold — it must be allowlisted with the FR-011 rationale, never "fixed."
- C-002: every gate error message must name the sanctioned seam so the developer knows where to route.
- NFR-001/002/003: composite key, concrete floor ≥ 38, shrink-only governance with pre-sweep baseline pinned.

### T001 — Composite-key allowlist machinery + YAML loader

**Purpose:** Implement the `(enclosing_qualname, token_line)` composite key used by both discriminators. This is net-new; no existing gate in this repo uses it.

**Files to create/edit:**
- Create `tests/architectural/test_resolution_authority_gates.py` (new, owns both discriminators)
- Create `tests/architectural/resolution_gate_allowlist.yaml` (the governance YAML; seeded in T004)

**Steps:**
1. Define a `GateAllowlistKey` datatype — a plain `NamedTuple` or `@dataclass(frozen=True)` with fields `enclosing_qualname: str` and `token_line: int`. This is the serialization target; `enclosing_qualname` is `ast.get_fullname` of the enclosing class/function chain (e.g. `MarkStatusCmd.run` or `module-level` for top-level code).
2. Write a `load_allowlist(path: Path) -> dict[str, list[GateAllowlistKey]]` loader. The YAML structure groups entries by gate name (keys `"canonicalizer"` and `"coord_authority"`); each entry has `qualname:` and `line:` scalars plus a mandatory `rationale:` string. The loader must reject entries missing a rationale (raise `ValueError` with a helpful message — no silent drift).
3. Write a `derive_live_key(node: ast.Call, tree: ast.Module) -> GateAllowlistKey` helper that walks the AST ancestors of `node` to determine the innermost enclosing function or class qualname. For top-level calls use `"<module>"`.
4. Write a `staleness_twin_guard(allowlist_keys: set[GateAllowlistKey], live_keys: set[GateAllowlistKey]) -> list[GateAllowlistKey]` that returns entries present in the allowlist but NOT in the live scan — these are stale entries that must fail the build.

**Validation checklist:**
- [ ] `GateAllowlistKey` is hashable and equality-comparable by `(qualname, line)` pair
- [ ] Loader raises `ValueError` on a YAML entry with a missing or empty `rationale`
- [ ] `derive_live_key` returns `"<module>"` for a call at file scope (no enclosing def/class)
- [ ] `staleness_twin_guard` returns an empty list when every allowlist entry has a matching live key

**Edge cases:** Nested functions must produce a dotted qualname chain (e.g. `outer.<locals>.inner`). Lambda expressions are treated as `<lambda>` per Python convention.

### T002 — Canonicalizer def-use discriminator

**Purpose:** Scan every call to `primary_feature_dir_for_mission` across `src/` and classify the handle argument as provably canonical (def-use from `_canonicalize_primary_read_handle` or from a known-canonical `feature_dir.name` in the same function) or as a violation. This is NOT a name-substring check; it is intra-function data-flow analysis at the AST level.

**Files:** `tests/architectural/test_resolution_authority_gates.py` (new functions in this module)

**Key source references:**
- `src/specify_cli/missions/_read_path_resolver.py:1212` — the blind primitive `primary_feature_dir_for_mission` (the call target to scan for)
- `src/specify_cli/missions/_read_path_resolver.py:1244` — `_canonicalize_primary_read_handle` (the canonical fold; the handle arg must flow from here)
- `src/specify_cli/missions/_read_path_resolver.py:454` — the bare probe INSIDE the fold; this IS a bare-handle call to the primitive and MUST be allowlisted with the C-001/FR-011 rationale, never "fixed"

**Steps:**
1. Write `scan_canonicalizer_call_sites(src_root: Path) -> list[tuple[Path, ast.Call, GateAllowlistKey]]` — AST-walks `src/**/*.py`, collects every `ast.Call` node where the callee is `ast.Name(id="primary_feature_dir_for_mission")` OR `ast.Attribute(attr="primary_feature_dir_for_mission")`. Handle both call forms. Include keyword-argument forms (e.g. `tasks.py:1346` passes the handle as a keyword arg — check both `node.args[1]` positional and a `feature_dir_name=` kwarg).
2. Write `is_def_use_canonical(handle_arg: ast.expr, enclosing_fn: ast.FunctionDef | ast.AsyncFunctionDef) -> bool` — walks the function body's assignment statements (via `ast.walk`) to check whether any `ast.Assign` or `ast.AnnAssign` in the SAME function assigns the handle name from a call to `_canonicalize_primary_read_handle`. Also accepts `feature_dir.name` attribute reads (the `.name` attribute of a `Path` that was returned by a resolver). "Same function" means the DIRECT enclosing `ast.FunctionDef`; do not look across function boundaries.
3. Write `check_canonicalizer_gate(src_root: Path, allowlist: set[GateAllowlistKey]) -> list[str]` that runs the scan, classifies each call site, and returns a list of violation strings for sites that are neither def-use-canonical nor in the allowlist. Each violation string must name `file:qualname:line` and the sanctioned seam (`_canonicalize_primary_read_handle`).

**Validation checklist:**
- [ ] A call `primary_feature_dir_for_mission(repo_root, raw_slug)` where `raw_slug` is never assigned from `_canonicalize_primary_read_handle` in the same function → classified as violation
- [ ] A call where `canonical_dir = _canonicalize_primary_read_handle(ctx, handle)` appears earlier in the same function and is passed as the handle → classified as canonical
- [ ] `feature_dir.name` as the handle arg → classified as canonical
- [ ] `_read_path_resolver.py:454` (the bare probe inside the fold) → NOT a violation because it is in the allowlist with C-001 rationale
- [ ] Keyword-arg call form `primary_feature_dir_for_mission(repo_root=r, dir_name=slug)` is detected correctly
- [ ] The scanner does not produce `False` violations for method calls where the callee is a dotted attribute of a different object (e.g. `self.resolver.primary_feature_dir_for_mission(...)`)

**Edge cases:** Some call sites inside the seam-internal modules (`_read_path_resolver.py` itself) use the primitive internally on an already-canonical argument — these must be allowlisted in T004 with rationale, not mis-classified as violations.

### T003 — Coord-authority discriminator

**Purpose:** Scan for mission-artifact WRITE sites that use the kind-blind `resolve_feature_dir_for_mission` where the kind-aware authority is mandated. The write-vs-read classification must be an explicit, documented predicate — not a one-entry tautology — and must not false-positive on the legitimate kind-blind coord-owned writes (`decisions/emit.py`, `widen/state.py`).

**Files:** `tests/architectural/test_resolution_authority_gates.py`

**Key source references:**
- `src/specify_cli/cli/commands/agent/tasks.py:1807` — the write leg that is the primary violator (routed in WP02)
- `src/specify_cli/decisions/emit.py` — legitimate coord-owned write that bypasses `commit_for_mission` by design (must be allowlisted, not flagged)
- `src/specify_cli/widen/state.py` — same; legitimate coord-owned write (allowlist)
- `src/specify_cli/agent/workflow.py` — `_commit_workflow_change` with `BookkeepingTransaction` (the model pattern)

**Steps:**
1. Define a `WRITE_INDICATOR_NAMES` constant — a frozenset of Python identifiers that, when used as the callee of a call in the same function that also calls `resolve_feature_dir_for_mission`, indicate a write operation. Examples: `"write_text"`, `"write_bytes"`, `"open"` (with mode `"w"`/`"wb"`/`"a"`), `"mkdir"`, `"rename"`, `"replace"`, `"shutil.copy"`, `"json.dump"`, `"yaml.dump"`, and any `commit_*` call. Document this set as the predicate in a module-level docstring or comment.
2. Write `scan_coord_authority_call_sites(src_root: Path) -> list[tuple[Path, ast.Call, GateAllowlistKey, bool]]` — AST-walks `src/**/*.py`, collects every call to `resolve_feature_dir_for_mission`. The `bool` return element indicates whether this site is classified as a write (True) or a read/probe (False) using the predicate from step 1. Writes that are in the allowlist (by `GateAllowlistKey`) are sanctioned; unsanctioned writes are violations.
3. Write `check_coord_authority_gate(src_root: Path, allowlist: set[GateAllowlistKey]) -> list[str]` returning violation strings for unsanctioned write sites. Each violation must name `file:qualname:line` and the kind-aware authority to use (`commit_for_mission(kind=)` / `resolve_planning_read_dir(kind=)`).

**Validation checklist:**
- [ ] `decisions/emit.py` call site is allowlisted and produces no violation
- [ ] `widen/state.py` call site is allowlisted and produces no violation
- [ ] `tasks.py:1807` (pre-WP02) classified as write and violation
- [ ] A pure read probe (`if resolve_feature_dir_for_mission(ctx).exists():`) does NOT trigger a violation
- [ ] The predicate is documented with a comment explaining what constitutes a "write" for this gate's purposes

**Edge cases:** `open()` with mode `"r"` is a read — do not classify it as a write. Assignment of the result to a local variable and subsequent filesystem operations require tracing the assignment, which is out of scope for Phase 1; instead, flag any call to `resolve_feature_dir_for_mission` in a function that contains at least one write indicator as a conservative classification and allow the allowlist to sanction legitimate ambiguous cases.

### T006 — Concrete floor + shrink-only staleness twin-guard

**Purpose:** Two enforcement mechanisms that prevent gate vacuity and allowlist inflation: (1) a concrete integer floor so a broken scanner producing zero rows trivially passes nothing, and (2) a staleness twin-guard that fails the build on any allowlist entry not matching a live call site.

**Files:** `tests/architectural/test_resolution_authority_gates.py`

**Steps:**
1. Write `test_canonicalizer_gate_floor` — run the full canonicalizer scan on the real `src/` tree (the full `src/specify_cli/` and `src/runtime/` subtrees), count discovered call sites (sanctioned + unsanctioned), and assert `count >= 38`. The literal `38` is the live census from the spec; do not use `> 0` or `>= 1` (NFR-002 explicitly rejects these).
2. Write `test_coord_authority_gate_floor` — same pattern; assert count ≥ a **hard-coded concrete integer literal floor** (e.g. `>= 12`). Run the write-candidate scan from T004 FIRST, then set the floor to the actual integer you counted — do NOT express the floor as `>= len(scanned_writes)` or any other self-referential expression (that is tautological and catches nothing). Live reality: there are 53 `resolve_feature_dir_for_mission` call sites and approximately 15 write-indicator files, so the honest write-candidate census is a dozen-plus. Set the floor to that concrete number.
3. Write `test_allowlist_no_stale_entries` — load `resolution_gate_allowlist.yaml`, derive the live `GateAllowlistKey` set from both scanners, call `staleness_twin_guard`, and assert the returned stale list is empty. This is the twin-guard specified by NFR-003.
4. Write `test_allowlist_shrink_only` — load the `canonicalizer_baseline` and `coord_authority_baseline` scalars from the YAML, derive the current allowlist entry counts, and assert `current_count <= baseline`. This is the enforcement of NFR-003's pre-sweep baseline: even after T004 seeds it, future WPs cannot add entries (they can only remove them as they route sites).

**Validation checklist:**
- [ ] `test_canonicalizer_gate_floor` would fail if the scanner returned an empty list (assert not satisfied by 0)
- [ ] `test_allowlist_no_stale_entries` fails if a YAML entry's `(qualname, line)` does not match any live call site — proving it detects stale entries
- [ ] `test_allowlist_shrink_only` would fail if a new entry were added to the YAML after seeding
- [ ] Both floor values are set to the actual live census count, not to `0`, `1`, or a rounded-down approximation

**Edge cases:** The `test_allowlist_no_stale_entries` test must re-derive live keys from the REAL `src/` tree (not from a stub), because the twin-guard must catch drift caused by actual source edits.

## Risks

- **Composite-key is net-new.** Do not under-budget T001; it is not a copy of any existing gate. The `(enclosing_qualname, token_line)` key requires AST ancestor traversal, which is absent from all existing Idiom-B precedents. Allow a full implementation pass before integration testing.
- **Floor miscalibration.** If the scanner is broken and produces 0 rows, a floor of `>= 38` still fails — but if the floor is set to `0` or `1`, it passes vacuously. The floor MUST be the live census number from the actual scan run. Run the scanner before writing the floor assertion.
- **Allowlist inflation anti-pattern.** The shrink-only guard prevents adding entries after seeding. If T004 discovers more than expected and the scanner is correct, accept the higher number as the honest baseline. Do not artificially cap it to hide violations.
- **coord-authority write-vs-read predicate.** There is no name proxy for "write" — the predicate must be explicit. A too-narrow predicate under-flags (leaves violations undetected); a too-broad one over-flags (false positives on `decisions/emit.py`). Test both the positive (write flagged) and negative (read not flagged) cases in T005.
- **C-001 merge-blocker.** If `_read_path_resolver.py:454` is NOT in the allowlist at merge, the gate will FAIL for the seam-internal probe. This must be seeded in T004 before T002's gate test runs against the real `src/` tree.
