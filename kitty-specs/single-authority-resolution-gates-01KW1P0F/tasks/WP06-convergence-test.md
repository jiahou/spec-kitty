## Objective

Write `tests/missions/test_resolution_convergence.py`: a stub-driven, parametrized
test that asserts the read-seam directory equals every write/placement-seam directory
for all five handle forms. The test is the executable proof of FR-006 (SC-005): after
WP02–WP05 route every call site through the canonical seam, no handle form can map
differently between the read leg and the write leg. The test is self-enforcing — it
rejects a constant-returning stub (T032) and requires a negative control proving it
was red under pre-fix code (T031).

No live `kitty-specs/` fixtures are used. This is a pure-stub test.

## Subtask Guidance

### T028 — Stub resolver with distinguishable P1–P5 per-form outputs

Build a `FakeResolver` class (or function) that implements the faithful P1–P5 cascade
documented in `_read_path_resolver.py`. The stub MUST return a **distinguishable output
per handle form** — each of the five forms maps to a unique fake directory name so the
test cannot pass vacuously. A constant-returning stub (one that always returns, e.g.,
`Path("/fake/missions/resolved")`) must be provably rejected by T032.

Suggested approach: define a `FORM_TO_DIR` mapping like:
```python
FORM_TO_DIR = {
    "full_slug": Path("/fake/resolved/by-full-slug"),
    "slug_mid8": Path("/fake/resolved/by-slug-mid8"),
    "bare_mid8": Path("/fake/resolved/by-bare-mid8"),
    "ulid":      Path("/fake/resolved/by-ulid"),
    "numeric":   Path("/fake/resolved/by-numeric"),
}
```
The stub's dispatch logic should mirror how the real seam disambiguates by form — pattern
match or regex on the handle string — not a single `return FAKE_DIR`. This distinguishable
mapping is the contract: T029 asserts `read_dir == write_dir` and both must come from the
stub in a non-constant way.

The stub must also be configurable to simulate ambiguous multi-match (for T030) and cold-
miss (for T030). A clean interface: pass `raise_ambiguous=True` or `raise_cold_miss=True`
to a factory that returns a configured stub.

Do NOT touch any live `kitty-specs/` dir or production call site.

### T029 — Parametrize all handle forms; assert read-seam ≡ write/placement-seam

Write a `@pytest.mark.parametrize` test over the five handle forms. For each form:

1. Call the read-seam resolution through the stub: obtain `read_dir`.
2. Call the write/placement-seam resolution through the stub: obtain `write_dir`.
3. Assert `read_dir == write_dir`.

Both legs must go through the same `FakeResolver` instance so the comparison is meaningful.
The `@pytest.mark.parametrize` ids should match the form names (`full_slug`, `slug_mid8`,
`bare_mid8`, `ulid`, `numeric`) so failures identify exactly which form diverged.

The test is stub-driven: it calls the stub's resolution function directly (bypassing the real
`primary_feature_dir_for_mission`) and asserts equal output. The point is that the same seam
logic applied to read and write returns identical results — if the real seam is uniform (after
WP02–WP05), the stub faithfully models that uniformity.

Mark with `@pytest.mark.fast` so it runs in the fast CI tier.

### T030 — Ambiguity-raise + cold-miss fail-closed cases

Add two dedicated test functions (not parametrized with the happy-path forms):

**Ambiguity case**: configure the stub to simulate a handle matching more than one mission.
Assert that the stub raises `MissionSelectorAmbiguous`. The assertion must use `pytest.raises`
and not catch the exception silently. This proves the convergence test does NOT hide
`MissionSelectorAmbiguous` behind a `try/except` wrapper.

**Cold-miss case**: configure the stub to simulate a handle for which no mission exists.
Assert that the stub raises a fail-closed exception (the real seam raises a
`MissionNotFoundError` or equivalent — use whatever is canonical in `_read_path_resolver.py`;
do NOT invent a new exception type). Assert with `pytest.raises`. Verify the raised
exception message does NOT contain a path that looks like a real on-disk location (no
verbatim passthrough of the handle into a `Path` join without a found-mission check).

Both cases must use the same `FakeResolver` factory from T028 with the appropriate flags set,
so the behavior is deterministic and unambiguous.

### T031 — Negative control: a pre-fix-divergent form (red-first proof)

Include ONE handle form that demonstrably mapped differently under pre-fix code — i.e.,
where the write leg used the kind-blind resolver and the read leg used the kind-aware
authority, producing different directories. This is the **red-first proof** (SC-005).

The negative control is a COMMENT-ANNOTATED parametrize case or a standalone test function
named `test_convergence_negative_control_pre_fix_divergence`. Its purpose is to document
that this exact form (e.g., `full_slug` under coordination topology) was the one that
diverged before WP02 (IC-04a) routed `mark_status`'s write leg through the kind-aware
authority.

In the negative control: show (as a comment or in an explicit sub-stub) what the two legs
returned under pre-fix code — they must be DIFFERENT values (not equal). Then assert that
with the fixed stub (post-WP02 behavior) they ARE equal. This proves the test was red before
the fix landed. Document the pre-fix divergence in a module-level docstring block.

The negative control does NOT re-introduce the old bad behavior in production code. It uses
the stub to simulate both the old behavior (distinguish read vs write output) and the fixed
behavior (same output), and asserts only the fixed behavior holds. A companion
`test_convergence_pre_fix_stub_diverges` function may use an explicit "old behavior" stub
variant to assert the old outputs differ — this is the red-first proof. Both functions must
pass with the correct stubs.

### T032 — Constant-stub-rejected guard

Add a test function `test_constant_stub_is_rejected` that instantiates a "bad" stub — one
that always returns the same `Path` value for every handle form regardless of input — and
asserts that the convergence test CANNOT pass with this stub. The guard makes the test
non-tautological.

Implementation pattern: the bad stub always returns `Path("/fake/constant")`. Run the same
five-form parametrize logic in a nested loop or call. Assert that all five calls return the
same value (i.e., the stub is constant). Then assert that the total set of return values has
length 1 (all equal). Then assert that this state is flagged as invalid by a
`is_constant_stub(stub)` predicate or equivalent check: the guard raises `ValueError` or
fails an assertion when the stub returns the same path for all handle forms.

This prevents a future developer from "fixing" a red test by making both legs return the same
constant — the guard detects it. The check is lightweight: iterate the five forms, collect
outputs, assert `len(set(str(out) for out in outputs)) > 1` or the stub's internal
`assert_distinguishable()` method. Document in the test docstring: "A constant-returning stub
cannot satisfy the convergence test — this guard proves it."

## Risks

- **Tautological stub**: if the stub returns a constant, both legs match trivially — T032 guards
  against this; verify the guard itself fails when given the constant stub.
- **Missing negative control**: the red-first proof (T031) is load-bearing for SC-005 and reviewer
  sign-off; a test without a negative control does not prove it was red before WP02.
- **Wrong exception type for cold-miss**: do not invent a new exception; check `_read_path_resolver.py`
  for the canonical cold-miss exception class used in the production seam.
- **Skipping the divergent cases**: T030 must not be skipped or `xfail`'d — they prove the seam
  is fail-closed and loud, not silent.
- **Over-scoping**: this WP creates ONE file in `tests/missions/`. Do not modify production code,
  `tasks.py`, or any architectural gate file.
