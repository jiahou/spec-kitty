# Tracer: Tooling Friction — Mission A (Common Docs Doctrine & Reconciliation)

> Standing-orders tracer (experiment #2095). Every place the tooling fought us. Seed at planning, append during implement, assess at close. Feeds the tooling-gap backlog.

## Seeded at planning (2026-06-27)

- **Contested primary checkout.** A concurrent `coord-authority-remediation-2160` implement loop ran on the *same* primary checkout and `git checkout`'d it out from under us mid-task (WP commit clobbered, branch switched). Resolution: moved all planning/implement work to the **`spec-kitty-doctrine-qol` sibling clone**. Lesson: parallel missions on one repo must use separate clones; the primary checkout is not safe to share with a live loop.
- **Doctrine dir naming (singular vs plural).** Initial WP02 `owned_files` + task prose used singular `src/doctrine/{directive,styleguide,tactic}/`; the real dirs are **plural** (`directives/styleguides/tactics/`). The glob matched zero files; caught by the post-tasks code-truth lens + would have mis-placed artifacts. Gap: no early validation that owned-file globs resolve.
- **`create_intent` required for new files.** `finalize-tasks` ownership validation rejects literal `owned_files` paths that match zero files unless they're declared in `create_intent`. Not obvious from the WP template; added post-hoc to WP01/03/04/05. Gap: the WP-prompt template should prompt for `create_intent` when owned_files are net-new.
- **`_inventory.py` hard-requires `citation_refs`.** Dropping `citation_refs` (WP04) requires editing `scripts/docs/_inventory.py`'s `_REQUIRED_KEYS` + the frozen `PageInventoryEntry` field — which crossed WP04's original ownership boundary. The squad caught it; WP04 now owns `_inventory.py`. Gap: schema-owning modules aren't surfaced when planning a field change.
- **`finalize-tasks` / `record-analysis` JSON output is rich-formatted.** Naive `json.load` on the piped output failed (`Expecting value: line 1 column 1`) — the rich panel precedes/wraps the JSON. Worked around by `grep`-ing the JSON fields. Gap: a `--json`-means-pure-json contract.
- **`map-requirements` normalizes WP frontmatter.** It reformats the YAML (reorders keys, adds `tracker_refs`/`tags`/`create_intent: []`) — fine, but surprised the diff.
- **`spec-kitty next` reports `discovery / not_started` for a fully-tasked mission.** The runtime mission state machine is separate from the agent-command planning; use `agent action implement WP##` directly for WP execution.

## Appended during implement
- WP01: no pre-flight/status friction (coord topology, clean lane flow).
- **`stale_analysis_report` blocks every WP claim after the first (the "analysis dance").** WP01 was claimed while the analysis was fresh; the WP01 *review* then forced filling `issue-matrix.md`, and I'd seeded the tracers — both committed AFTER the analysis. The implement gate (`agent action implement WP02…`) then rejects with `Error: analysis_report_required … Reason: stale_analysis_report`. The freshness is invalidated by **any** mission-artifact change after the analysis (issue-matrix, tracers), not just spec/plan/tasks. Worse: **`record-analysis` is idempotent** — re-running it with identical findings produces a byte-identical `analysis-report.md`, so `git` sees "no changes", no fresher commit is produced, and the staleness persists no matter how many times you re-record (confirmed 3×). Gaps: (a) the freshness check should ignore non-spec/plan/tasks artifacts; (b) the canonical order is *analyze AFTER issue-matrix fill*, but the issue-matrix is only forced-complete at WP-**review** (after analyze runs) — a chicken-and-egg the workflow doesn't surface; (c) `record-analysis` needs a `--force`/re-snapshot to mint a fresh artifact on demand. Workaround under test: force a genuinely-new analysis-report commit.
- **Tracers on the PLANNING branch fight the per-WP + per-analysis guards (operator insight, cfr #2160).** Two concrete frictions: (1) WP05's `move-task` pre-flight blocked on the uncommitted mission tracer in the shared planning checkout — the reviewer had to commit `design-decisions.md` to clear a *code* WP's approval gate; (2) every tracer commit re-stales the analysis (pushes `analysis-report.md` off the latest-commit position the sha-freshness guard wants). Root: a **mission-wide, append-during-implement** artifact lives on a **per-mission-planning** surface that both the per-WP pre-flight and the analysis-freshness guard scan. Right model (mirrors #2160's coordination-authority): tracers are **coordination-level state**, so their home should be **topology-dependent — the coordination branch (coord topology) / the primary-or-mission branch (flat)** — NOT the planning branch. Feeds #2095 (tracer experiment must specify *where*) + #2160.
- **The merge SQUASH-clobbered the planning-branch tracer edits (cfr #2160 split-brain, witnessed at close).** `spec-kitty merge` squash-merged the mission branch onto `docs/2165-consolidation-research`; the mission-branch's older tracer copy overwrote the two latest planning-branch tracer notes (the skills-as-doctrine design note + the placement note above). Had to be re-applied post-merge. This is the *same* coord/planning split-brain biting the tracers a third time — strong evidence the tracer home must follow coordination-authority (#2160), not the planning branch. Also: acceptance-matrix + the WP06 cycle-3 review artifact each had to be written to BOTH the planning checkout AND the `-coord` worktree to satisfy accept/merge gates (the gates read the coord worktree, not the planning branch the implement loop writes to).
