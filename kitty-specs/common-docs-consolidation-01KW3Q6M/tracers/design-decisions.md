# Tracer: Design Decisions — Mission A (Common Docs Doctrine & Reconciliation)

> The rationale that would otherwise evaporate. Seed at planning, append during implement, assess at close. (The authoritative record is the reconciliation ADR; this tracer captures the *why behind the why*.)

## Seeded at planning (2026-06-27)

- **Candidate A — in-file frontmatter is the metadata SSOT; the page-inventory becomes a generated lockfile; `citation_refs` dropped.** Why over Candidate B (sidecar SSOT): the live DocFX site reads frontmatter, not the sidecar (generator-native); the standard mandates in-file; it retires the `LEAK-FRONTMATTER-MISMATCH` split-brain *by construction* (one copy of the datum). B would perpetuate the very split-brain the consolidation exists to kill.
- **`adr/<era>/` (deviate from the standard's flat `adr/`).** Why: 117 unique ADRs across three eras carry era-of-decision reasoning; flattening erases version-stance the public site is built around. The deviation preserves history; the *living* design unifies unversioned (era belongs to history, not current state).
- **`doc_status` (not bare `status`).** Why: bare `status` collides with the WP-lane status model (terminology-canon).
- **DocFX redirect = generated `<meta http-equiv=refresh>` stubs.** Why: DocFX + GitHub Pages have no native alias/redirect and no server rules; a client-side meta-refresh stub is the only mechanism that resolves on static Pages.
- **Glossary read-path (the load-bearing C-001 call).** The dashboard reads `.kittify/glossaries/<scope>.yaml` **seed files** via `load_seed_file()` — NOT the human `glossary/contexts/*.md`. So moving the human markdown to `context/` must preserve/regenerate the seed read-path; the doctrine-extraction source is the seed. (Verified live during WP01: `src/glossary/scope.py:119`.)
- **The directive→ratchet binding must RESOLVE, not just appear.** Why: a hardcoded `directive_ref = "<id>"` string passes a naive "string appears" test while being an orphan. The binding self-test asserts the id resolves to a loaded directive (a node in `graph.yaml` / via DoctrineService), sourced from a single shared constant. This is the least-defended-but-load-bearing thesis ("doctrine is bound, not advisory").
- **Rulers report-only in A, blocking in B.** Why: a gate first enforced against an uncleaned tree can't honestly police it (gate-unmask-cannot-self-validate); measure the baseline in A, enforce against the cleaned tree in B.

## Appended during implement
- **Skills ship as doctrine tactics, not installed Agent Skills (operator decision, 2026-06-27).** WP06's review caught a real conflict: the ADR's Neutral note said the 3 Common Docs skills "install as peer skills", but the implementer made them out-of-scope citing the research. The operator chose a third path — **implement them as doctrine artifacts (tactics), via doctrine-daphne**. Why: their value is *guidance, not automation* (the standard automates nothing load-bearing), so doctrine — alongside the directive/styleguide — is the coherent home, and `find` uses the DRG + inventory rather than a static table (matching the ADR's one firm sub-decision). **Vehicle = re-open WP02** (not a new WP), because the skill-tactics must wire into the generated `graph.yaml`, which only one WP can cleanly own; a separate WP would collide on the DRG at merge.
- **The ADR's "install as peer skills" note is deferred to Mission B (operator-confirmed).** Fixing it in-place would mean re-opening the WP01 spine and cascading an ADR re-merge through every lane (each carries a copy). Mission B re-touches the ADR anyway, so the note is reconciled there; WP06's citation is honest in the interim. Cost: a documented, temporary ADR-vs-impl mismatch bounded to Mission A's lifetime.
